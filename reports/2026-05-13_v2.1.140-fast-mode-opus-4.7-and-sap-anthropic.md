## Layer 1: 公式チェンジログ — v2.1.140 (2026-05-12 21:09 UTC)

**位置づけ**: 2026-05-11 18:43 UTC の v2.1.139 大型機能リリース（Agent View / `/goal` / バックグラウンドセッション機能群追加、50+ バグ修正）から **約 26 時間後のフォローアップリリース**。v2.1.139 で投入された新機能の実運用で発覚した issue を集中修正。新機能追加なし。

### Agent tool / Subagent 改善

#### Agent tool `subagent_type` マッチング改善（case- / separator-insensitive）✅

- **変更**: `Task` ツール（subagent 起動）の `subagent_type` パラメータが **大文字小文字・区切り文字を区別しないマッチング**を採用
- **例**: `"Code Reviewer"` / `"code-reviewer"` / `"code_reviewer"` / `"CodeReviewer"` のいずれも同じ subagent（実 ID `code-reviewer`）を解決
- **背景**: v2.1.139 で Agent View / Agent Teams / `--agent` フラグが拡充された結果、ユーザーが subagent 名を自由表記で記述するケースが増加。**カノニカル ID（lowercase + hyphen-separated）の機械的記法を強制すると、対話的な自然言語呼び出しと衝突**するため、入力許容度を引き上げた
- **影響**: Agent View や `@agent-name` メンション、`--agent <name>` フラグ、hook の `subagent_type` 指定で「typo に近い揺れ」が許容されるようになり、プロンプト経由の subagent 起動の摩擦が低下
- **情報源**: [GitHub Releases v2.1.140](https://github.com/anthropics/claude-code/releases)

#### Agent カラーパレット更新 ✅

- Agent View / Agent Teams で使用される agent 識別色のパレットを更新。視認性向上が目的（具体的な色変更内容はチェンジログ未公表）
- v2.1.139 で投入された Agent View の `state` 表示色（✽ Working / 🟡 Needs input / Dimmed Idle / 🟢 Completed / 🔴 Failed / ⚫ Stopped）とは別レイヤーの「per-agent identity color」を指すと推定

### `/goal` コマンド修正

#### `/goal` が `disableAllHooks` / `allowManagedHooksOnly` 設定下で silently ハングする問題修正 ✅

- **症状**: managed settings で `disableAllHooks: true` または `allowManagedHooksOnly: true` を設定すると、`/goal` の completion check に必要な Stop-hook が登録されず、**ターン完了後にゴール検証が走らず、ステータスインジケータが永久に "evaluating goal..." 表示のまま停止**
- **修正**: Hook が無効化されている検出時に **明確なエラーメッセージを表示**するように変更（silent hang から explicit refusal へ）
- **エンタープライズ影響**: Hook を全社一律で制限している組織で `/goal` が事実上利用不可だった問題が解消（エラーメッセージで利用不可を明示することで、管理者 / 利用者間のサポート摩擦を低減）
- v2.1.139 で `/goal` 投入時に **runaway guard デフォルト 500 回 Stop-hook continuation** が設計されていたため、Stop-hook が登録できない環境では goal evaluation 自体が成立しない構造的問題だった

### バックグラウンドセッション / `claude --bg` 修正

#### `claude --bg` が "connection dropped mid-request" で失敗する問題修正 ✅

- **症状**: バックグラウンドセッションのホスト supervisor プロセスが **idle-exit 直前**（v2.1.139 で導入された「アイドル 1 時間で supervisor process 停止」挙動）に新規 `claude --bg` リクエストが到着すると、supervisor が応答途中で shutdown → "connection dropped mid-request" エラーで起動失敗
- **修正**: クライアント側で supervisor の idle-exit を検出して **自動再起動 + リトライ**するように変更
- **背景**: v2.1.139 で導入された supervisor プロセスのライフサイクル管理（attach で再起動、idle-exit、binary update 検出再起動、マシンスリープ復元）のうち、idle-exit のレースコンディションを修正

#### バックグラウンドサービス起動失敗の修正（エンタープライズ endpoint security 対応）✅

- **症状**: CrowdStrike Falcon / SentinelOne / Carbon Black など **エンタープライズ endpoint security 製品**を導入したマシンで supervisor process 起動が timeout により失敗
- **修正**: 起動 timeout を延長し、endpoint security 製品によるバイナリ scan / sandbox 検証時間を考慮した起動シーケンスに変更
- **エンタープライズ影響**: 5/11 v2.1.139 リリース直後に大企業 IT 部門から「バックグラウンドモードが起動しない」という報告が複数寄せられた可能性（チェンジログから推定）

### 設定 hot-reload / Remote managed settings 修正

#### Symlinked settings file の hot-reload regression 修正 ✅

- **症状**: `~/.claude/settings.json` を symlink（dotfiles 管理ツール chezmoi / yadm 利用時に多発）で配置している環境で、settings file の編集が **誤って "ConfigChange hook" を spurious に発火**したり、変更検出がそもそも走らなかったりする regression
- **修正**: symlink target を正しく resolve して inotify / FSEvent を貼るように変更
- v2.1.139 でも類似の symlink fix が含まれていたが、その対応で別の regression が混入していたものを修正

#### Remote managed settings が 401 でリトライしない問題修正 ✅

- **症状**: enterprise managed settings の取得（例: `forceRemoteSettingsRefresh: true`）で 401 Unauthorized 応答を受けた場合、**fail-closed でクラッシュ**してリトライしない
- **修正**: 401 受信時に **force-refreshed token で 1 回リトライ**するロジック追加
- **背景**: SSO token 期限切れタイミングと remote settings refresh のタイミングが重なる「ほぼランダムに見える起動失敗」を再現安定化

### その他バグ修正

| 修正項目 | 詳細 |
|---------|------|
| `/loop` が背景タスクで redundant wakeup スケジューリング | 同じターゲット時刻に複数 wakeup を重ねる問題を修正 |
| Windows: missing executables による synchronous spawn → event-loop stall | 存在しないコマンドの起動試行が同期的に走り、TUI が数秒固まる recurring stall を修正 |
| `Read` ツール: `offset` の whitespace-padded / `+`-prefixed string が validation 失敗 | `" 100"` / `"+100"` を `100` として受容するように修正 |
| ネイティブターミナル: ターミナルがフォーカス喪失時に入力カーソルが入力 caret に留まらない | カーソル位置が壊れる UI バグを修正 |
| プラグイン: デフォルトコンポーネントフォルダ（例: `commands/`）が `plugin.json` で同名キー設定により silently 無視 | 無視されている場合に warning を出すように変更 |

## Layer 2: 公式プラットフォーム — Fast mode for Opus 4.7（2026-05-12）

### Fast mode が Claude Opus 4.7 に対応 🔬

**発表媒体**: [Claude Platform Release Notes (5/12)](https://platform.claude.com/docs/en/release-notes/overview) / [Fast mode (Docs)](https://platform.claude.com/docs/en/build-with-claude/fast-mode) / [What's new in Claude Opus 4.7 (Docs)](https://platform.claude.com/docs/en/about-claude/models/whats-new-claude-4-7)

#### 概要

- **リリース状態**: 🔬 **研究プレビュー継続**（Opus 4.6 fast mode は 2026-02-07 launch、5/12 で Opus 4.7 サポート追加）
- **アクティベーション**: API リクエストで以下 3 つを同時設定
  - `speed: "fast"`
  - `model: "claude-opus-4-7"`
  - beta header: `fast-mode-2026-02-01`
- **パフォーマンス**: 標準 Opus 4.7 比で **最大 2.5x の output token throughput**
- **対象プラン / アクセス**:
  - **Anthropic Console API**: waitlist 制（[claude.com/fast-mode](https://claude.com/fast-mode) で申請）
  - **Claude subscription plans**: extra usage 残高経由でアクセス可能

#### 価格

- **入力 $30 / MTok、出力 $150 / MTok**（標準 Opus 4.7 の **6 倍**）
- **Opus 4.6 fast mode と同一価格**（Opus 4.6 でも $30 / $150 per MTok）
- **prompt caching multiplier・data residency multiplier は fast mode pricing の上に積層適用**

#### レート制限・アクセス条件

- Opus 4.6 fast mode と同一のレート制限・waitlist 機構を継承（Anthropic は「availability is limited while we gather feedback」と明記）
- 1M context window との互換性: Opus 4.7 は 1M context が GA（beta header 不要）だが、**fast mode 利用時の 1M context との組み合わせ可否は公式ドキュメント未明示**（5/13 時点）

#### Claude Code との関係

- **Claude Code 内の `/fast` トグルは現状 Opus 4.6 fast mode を対象**（CURRENT_FEATURES.md 既存記述）
- Claude Code セッションから Opus 4.7 fast mode を利用するパスは **5/13 時点で公式 changelog / Help Center に記載なし**（API 直接呼び出しとして使うか、Claude Code が今後 Opus 4.7 fast mode 対応する可能性は高いが未確認）
- **Pro / Max ユーザーの compute crunch ナラティブ（5/8 Fortune Amodei 公式コメント「avg developer 20 hours/week with Claude Code」）への解の一つ**として位置付け可能 — 同じ Opus 4.7 品質を 2.5x の token throughput で消化可能になり、5 時間 rate-limit window 内の生産性が実質倍増する余地

#### 戦略的位置づけ

- **Opus 4.7 リリース（2026-04-16 GA）から約 4 週間後の fast mode 対応**は、Opus 4.6 リリース（2026-02-05）からの fast mode 対応（2026-02-07、2 日後）と比べて長い間隔だった点に注意
- 推測: ① Opus 4.7 の新しい tokenizer（4.6 から API breaking change）/ xhigh effort level / 高解像度ビジョン（長辺 2,576px）対応により、fast mode の推論最適化が複雑化、② 5/6 SpaceX Colossus 1（300MW / 220K GPU）/ 5/8 Akamai $1.8B / 5/11 Claude Platform on AWS GA でコンピュート供給確保が完了した直後のタイミングであり、**コンピュート確保 → fast mode 拡張という資源優先順位**で説明可能
- Anthropic にとっては「同じ 1 GPU からのトークン生成 throughput を 2.5x で売る」高利益率商品であり、IPO 前のマージン改善材料としても寄与

## バグ修正・パフォーマンス改善（v2.1.140 まとめ）

### 重要修正

- **`/goal` ハング修正**: `disableAllHooks` / `allowManagedHooksOnly` 設定下で `/goal` がエラーを返さず永久に "evaluating goal..." 表示で固まる問題を、明確なエラーメッセージ表示に変更（エンタープライズ Hook 制限環境での silent failure 解消）
- **`claude --bg` connection drop 修正**: バックグラウンド supervisor process の idle-exit レースで "connection dropped mid-request" を出していた問題を、クライアント側の自動リトライで解決
- **エンタープライズ endpoint security 起動 timeout 延長**: CrowdStrike Falcon / SentinelOne / Carbon Black 等の endpoint security 製品環境での supervisor process 起動失敗を解消
- **Remote managed settings 401 retry**: SSO token 期限切れタイミングと remote settings refresh の重なりによる起動失敗を、force-refreshed token での 1 回リトライで解消

### UI / 入力修正

- ネイティブターミナルのフォーカス喪失時にカーソルが入力 caret に留まらない問題修正
- `Read` ツールが `offset` パラメータの whitespace-padded（`" 100"`）/ `+`-prefixed（`"+100"`）string を受容するように

### Windows / プラグイン修正

- Windows: missing executables による synchronous spawn → event-loop stall 修正
- プラグイン: `plugin.json` で commands/agents/skills のキーを設定した場合に、対応する default folder が silently 無視されていた問題を warning 表示で可視化

### Settings / hot-reload 修正

- Symlinked `~/.claude/settings.json` の hot-reload regression 修正（dotfiles 管理ツール ユーザー向け）
- `/loop` の redundant wakeup スケジューリング修正
